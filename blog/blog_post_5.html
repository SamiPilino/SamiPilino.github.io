Wednesday, September 20, 2023

Worked on variations of the timer project all day.

Minor Improvements to the Granola Timer

I started with some minor improvements to the granola timer based on what I remember from the time I recorded the process with the initial timer project. This included removing the session summarization textbox (which is not needed for sufficiently well-defined processes), automatically populating the summary field with session information instead, and .

Finally Implemented Timer Audio (Yay! Interrupts actually work now!)

I also went through some WAV files of different UI sound effects and found some good "focus timer", "relaxation", and "alarm" sounds I could use, and added those to all the different timer projects.

Scrum Timer Attempt

I created a "Scrum" timer from a copy of the Daily Schedule timer and started trying to outline a Sprint in timer form. I opened up a temporary brainstorming file for helping sketch out my ideas and added a ProductGoal, ProductBacklog, SprintGoal, and SprintBacklog. As I sketched out the basic ideas, adding comments and updating my notes along the way, I started realizing I needed to figure out the more basic task-management logic first before adding these more useful artifacts (backlogs, increments, etc.) to the process. So, I paused development on the Scrum Timer project and created an RTOS Timer, which will focus on the core task-management logic for running tasks, scheduling them, and handling interrupts.

RTOS Timer

To begin the RTOS timer, I designed a simple "report, context-switch, take a break, work" loop which forms the basis for more real-time operation. The original countdown timer is used as the SysTick interrupt, which periodically reschedules tasks so that no one task can hog the CPU. Next, I added a Dashboard area for displaying session-specific components such as a list of tasks or information about the current task. I added a list of sample tasks and defined their "stack frames" with instructions, an instruction pointer, and stack/heap space for any other memory such as variables and functions used by the instructions. Then, I added some task info components to the Dashboard so we can see information about the current task during the "Running Task" stage, but I had to make the code more readable so that I could more easily fix an issue where the Dashboard wasn't updating properly because I had mixed in some application logic during the prototyping stages of creating the Dashboard components. Turns out, I needed to run only one of those new update functions whenever its data changed instead of updating the entire UI. Finally, I added "Block" and "Yield" buttons (which don't do anything yet) to the Current Instruction section after the "Next Instruction" button and started getting stumped on what I should do next. So, I added a "status" field to the tasks for future use in the scheduling algorithm and spent some time brainstorming ideas for the rest of the RTOS Timer implementation.
